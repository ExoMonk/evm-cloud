# Variable Reference

All configuration variables for evm-cloud, organized by category. Variables marked **sensitive** are redacted in Terraform plan/apply output.

## Most Impactful Variables

These are the variables that most affect your deployment's cost, performance, and architecture:

| Variable | Impact | Default | What It Controls |
|----------|--------|---------|-----------------|
| `compute_engine` | Architecture | `ec2` | Entire deployment model: EC2+Docker, EKS, k3s, or bare metal |
| `ec2_instance_type` / `k3s_instance_type` | Cost + Performance | `t3.small` | Instance size — biggest cost driver |
| `ec2_indexer_mem_limit` | Performance | `2g` | How much RAM rindexer can use (increase for large indexing jobs) |
| `workload_mode` | Workflow | `terraform` | Whether Terraform or external tools manage workloads |
| `postgres_instance_class` | Cost | `db.t4g.micro` | RDS instance size (when using managed Postgres) |
| `network_enable_nat_gateway` | Cost | `false` | NAT gateway adds ~$35/mo — skip for dev |

| Variable | Type | Default | Description |
|----------|------|---------|-------------|
| `networking_enabled` | `bool` | `false` | Enable VPC provisioning. Required for most AWS deployments. |
| `network_environment` | `string` | `"dev"` | Profile: `dev` (minimal SGs), `production` (multi-AZ, NAT), `platform` (full). |
| `network_vpc_cidr` | `string` | `"10.42.0.0/16"` | VPC CIDR block. |
| `network_availability_zones` | `list(string)` | `["us-east-1a", "us-east-1b"]` | AZs for subnet placement. |
| `network_enable_nat_gateway` | `bool` | `false` | NAT gateway for private subnet egress. **Adds ~$35/mo.** Skip for dev. |
| `network_enable_vpc_endpoints` | `bool` | `false` | S3 gateway + interface endpoints (ECR, CloudWatch, SSM). |
| `aws_region` | `string` | `"us-east-1"` | AWS region. |
| `aws_skip_credentials_validation` | `bool` | `false` | Skip credential checks (for LocalStack testing). |

### Security Groups

The networking module creates role-based security groups. Each compute instance gets its primary SG plus the `indexer` identity SG for database access.

| Security Group | Purpose | Attached To |
|---------------|---------|-------------|
| `ec2-sg` | SSH, eRPC port, full egress | EC2 instances (`compute_engine = "ec2"`) |
| `k3s-sg` | SSH, k3s API (6443), Flannel (8472), Kubelet (10250), NodePort range | k3s server and worker instances |
| `indexer-sg` | Database access: Postgres (5432), ClickHouse HTTP (8123), ClickHouse native (9000) | All compute instances (EC2, k3s server, k3s workers) |
| `database-sg` | Accepts ingress from `indexer-sg` only | RDS PostgreSQL instances |
| `erpc-sg` | Accepts ingress from cluster ingress/controller path and `indexer-sg` | EKS eRPC pods |

The `indexer-sg` acts as a portable identity — the `database-sg` ingress rules reference it, so any instance with `indexer-sg` attached can reach the database regardless of compute engine.

## k3s Compute

Required when `compute_engine = "k3s"`.

| Variable | Type | Default | Sensitive | Description |
|----------|------|---------|-----------|-------------|
| `k3s_instance_type` | `string` | `"t3.small"` | No | EC2 instance type for the k3s host. |
| `k3s_version` | `string` | `"v1.30.4+k3s1"` | No | Pinned k3s version. |
| `ssh_private_key_path` | `string` | `""` | Yes | Path to SSH private key file used for provisioning and config updates. Required for EC2, K3s, and bare metal deployments. |
| `k3s_api_allowed_cidrs` | `list(string)` | `[]` | No | CIDRs allowed to access k3s API (port 6443). Defaults to VPC CIDR. **Add your IP for Terraform provisioning.** |
| `k3s_worker_nodes` | `list(object)` | `[]` | No | Worker nodes to join the k3s cluster. See below. |
| `ssh_public_key` | `string` | `""` | Yes | SSH public key for the k3s EC2 key pair. |

> **Important:** `k3s_api_allowed_cidrs` must include your local IP (or `0.0.0.0/0` for dev) so Terraform can SSH into the instance during provisioning.

### k3s Worker Nodes

Add worker nodes to create a multi-node k3s cluster. Each worker gets a dedicated EC2 instance with a role label for pod scheduling.

```hcl
k3s_worker_nodes = [
  { name = "backfill", role = "indexer", instance_type = "t3.small", use_spot = true },
]
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `name` | `string` | (required) | Unique name for the worker node. |
| `role` | `string` | `"general"` | Node label `evm-cloud/role=<value>`. Values: `indexer`, `database`, `evm-node`, `monitoring`, `general`. |
| `instance_type` | `string` | `"t3.small"` | EC2 instance type for this worker. |
| `use_spot` | `bool` | `false` | Use a spot instance (~70% cheaper). Recommended for interruptible workloads like backfill indexers. |
| `host` | `string` | (auto) | For bare_metal provider: the worker's IP/hostname. For AWS: auto-provisioned. |

When `k3s_worker_nodes = []` (default), only the server node is created — backward compatible with existing single-node deployments.

See the [k3s multi-node example](./examples/k3s-multi-clickhouse) for a complete walkthrough.

## Indexer (rindexer)

| Variable | Type | Default | Description |
|----------|------|---------|-------------|
| `indexer_enabled` | `bool` | `false` | Enable rindexer deployment. |
| `indexer_image` | `string` | `"ghcr.io/joshstevens19/rindexer:latest"` | Container image. |
| `indexer_rpc_url` | `string` | `""` | RPC endpoint URL. Auto-resolves to eRPC internal URL when both are enabled. |
| `indexer_storage_backend` | `string` | `"postgres"` | Storage: `postgres` (managed RDS) or `clickhouse` (BYODB). |
| `rindexer_config_yaml` | `string` | `""` | Full rindexer.yaml content. Use `$${RPC_URL}` and `$${DATABASE_URL}` for runtime interpolation (double `$$` is Terraform's escape for literal `${}`). |
| `indexer_postgres_url` | `string` | `""` | PostgreSQL connection string (e.g. `postgres://user:pass@host:5432/db`). Used for `bare_metal` + `postgres` deployments. |
| `rindexer_abis` | `map(string)` | `{}` | ABI files: `{ "ERC20.json" = file("abis/ERC20.json") }`. |
| `indexer_instances` | `list(object)` | `[]` | Multiple indexer instances with independent configs. Each becomes a separate Helm release. See below. |

> **Tip:** When both `rpc_proxy_enabled` and `indexer_enabled` are true, the indexer's RPC URL automatically points to the eRPC internal service — no manual wiring needed.

### Multi-Instance Indexer

Deploy multiple rindexer instances (e.g., live + backfill) with independent configs and node scheduling:

```hcl
indexer_instances = [
  { name = "indexer", node_role = "server" },
  { name = "backfill", config_key = "backfill", node_role = "indexer" },
]
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `name` | `string` | (required) | Instance name. Becomes Helm release `<project>-<name>`. |
| `config_key` | `string` | (none) | Subdirectory in config bundle for this instance's `rindexer.yaml`. If unset, uses the root config. |
| `node_role` | `string` | (none) | Pin to nodes with `evm-cloud/role=<value>` label via `nodeSelector`. |
| `workload_type` | `string` | `"deployment"` | Kubernetes workload type: `"deployment"` (default, long-running) or `"job"` (one-shot backfill, exits on completion). |

When `indexer_instances = []` (default), a single indexer release is deployed — backward compatible.

For one-shot backfill indexers, set `workload_type = "job"` in the instance definition:

```hcl
indexer_instances = [
  { name = "indexer", node_role = "server" },
  { name = "backfill", config_key = "backfill", node_role = "indexer", workload_type = "job" },
]
```

The `"job"` workload type renders a Kubernetes Job instead of a Deployment — the pod runs to completion and does not restart.

## Secrets Management

Control how credentials reach workloads. See the [Secrets Management guide](./guides/secrets-management.mdx) for architecture details.

| Variable | Type | Default | Sensitive | Description |
|----------|------|---------|-----------|-------------|
| `secrets_mode` | `string` | `"inline"` | No | How secrets reach workloads: `inline`, `provider` (AWS SM + ESO), or `external` (user-managed store). |
| `secrets_manager_secret_arn` | `string` | `""` | Yes | ARN of a pre-existing Secrets Manager secret. When set with `secrets_mode = "provider"`, skips secret creation (BYOA). |
| `secrets_manager_kms_key_id` | `string` | `""` | No | KMS key ID or alias for Secrets Manager encryption. Omit for AWS default key. |
| `external_secret_store_name` | `string` | `""` | No | Name of a user-managed `ClusterSecretStore`. Required when `secrets_mode = "external"`. |
| `external_secret_key` | `string` | `""` | No | Secret key/name in the external store that holds workload env vars. Required when `secrets_mode = "external"`. |
| `eso_chart_version` | `string` | `"0.9.13"` | No | Pinned External Secrets Operator Helm chart version. |

### Constraints

- `secrets_mode = "provider"` requires `infrastructure_provider = "aws"` (uses AWS Secrets Manager + IAM instance profile)
- `secrets_mode = "external"` requires both `external_secret_store_name` and `external_secret_key` to be set
- `bare_metal_secrets_encryption = "sops_age"` is not yet implemented (T2 deferred)

## Ingress & TLS

Control how traffic reaches your services with automatic TLS termination. See the [TLS & Ingress guide](./guides/tls-ingress.mdx) for setup instructions.

| Variable | Type | Default | Sensitive | Description |
|----------|------|---------|-----------|-------------|
| `ingress_mode` | `string` | `"none"` | No | TLS termination mode: `none`, `cloudflare` (recommended), `caddy`, or `ingress_nginx`. |
| `erpc_hostname` | `string` | `""` | No | Public hostname for eRPC TLS certificates (e.g. `rpc.example.com`). Required when `ingress_mode != "none"`. |
| `ingress_tls_email` | `string` | `""` | No | Email for Let's Encrypt ACME registration. Required for `caddy` and `ingress_nginx` modes. |
| `ingress_cloudflare_origin_cert` | `string` | `""` | Yes | Cloudflare origin certificate (PEM). Required for `cloudflare` mode. |
| `ingress_cloudflare_origin_key` | `string` | `""` | Yes | Cloudflare origin certificate private key (PEM). Required for `cloudflare` mode. |
| `ingress_cloudflare_ssl_mode` | `string` | `"full_strict"` | No | Cloudflare SSL/TLS mode: `full` or `full_strict`. |
| `ingress_tls_staging` | `bool` | `false` | No | Use Let's Encrypt staging CA (for testing — avoids rate limits). |
| `ingress_hsts_preload` | `bool` | `false` | No | Add `preload` directive to HSTS header. Only enable after confirming HSTS preload eligibility. |
| `ingress_request_body_max_size` | `string` | `"1m"` | No | Maximum request body size. |
| `ingress_caddy_image` | `string` | `"caddy:2.9.1-alpine"` | No | Caddy Docker image (EC2/bare_metal only). |
| `ingress_caddy_mem_limit` | `string` | `"128m"` | No | Caddy container memory limit. |
| `ingress_nginx_chart_version` | `string` | `"4.11.3"` | No | ingress-nginx Helm chart version for `ingress_nginx` mode (automated in k3s deployer). |
| `ingress_cert_manager_chart_version` | `string` | `"1.16.2"` | No | cert-manager Helm chart version for `ingress_nginx` mode (automated in k3s deployer). |

### Ingress Mode by Compute Engine

| Mode | EC2 / Bare Metal | k3s | EKS |
|------|-----------------|-----|-----|
| `none` | No TLS | No TLS | No TLS |
| `cloudflare` | Caddy + CF origin cert | ingress-nginx + CF TLS secret | Supported by guardrails; deployer automation may be engine-specific |
| `caddy` | Caddy + Let's Encrypt ACME | Not supported (use `ingress_nginx`) | Not supported (use `ingress_nginx`) |
| `ingress_nginx` | Not supported (use `caddy`) | ingress-nginx + cert-manager | Supported by guardrails; install ingress/cert-manager via your EKS workflow |

### Constraints

- `erpc_hostname` is required when `ingress_mode != "none"`
- `cloudflare` mode requires both `ingress_cloudflare_origin_cert` and `ingress_cloudflare_origin_key`
- `caddy` mode requires `ingress_tls_email` and is only available on EC2/bare_metal
- `ingress_nginx` mode requires `ingress_tls_email` and is available on `k3s` and `eks`

### Alert Rules (built-in)

When monitoring is enabled, these PrometheusRule alerts are created automatically:

| Alert | Condition | Severity |
|-------|-----------|----------|
| RindexerHighLag | `rindexer_blocks_behind > 100` for 5m | warning |
| RindexerCriticalLag | `rindexer_blocks_behind > 1000` for 10m | critical |
| RindexerDeepReorg | `rindexer_reorg_depth > 2` for 1m | warning |
| RindexerFrequentReorgs | `>5 reorgs` in 15m | warning |
| RindexerRPCErrorRate | `>5%` RPC errors for 5m | warning |
| RindexerDBErrors | Any DB write errors in 5m | critical |
| RindexerDown | Target unreachable for 2m | critical |

### Dashboards (built-in)

Three Grafana dashboards are provisioned automatically:

| Dashboard | Panels |
|-----------|--------|
| **rindexer** | Blocks behind, events/sec, RPC error rate, DB ops/sec, reorgs detected, reorg depth |
| **eRPC** | Request rate, error rate, upstream latency (p99), cache hit rate, upstream health |
| **Infrastructure** | CPU by pod, memory by pod, network I/O, disk usage, pod restarts, node CPU |

## Instance Sizing

### EC2 / k3s Host

| Instance | vCPU | RAM | Monthly Cost | Use Case |
|----------|------|-----|-------------|----------|
| `t3.micro` | 2 | 1 GB | ~$8 | Testing only (eRPC barely fits) |
| `t3.small` | 2 | 2 GB | ~$16 | Light dev (1 small indexer) |
| `t3.medium` | 2 | 4 GB | ~$33 | Fits eRPC (1G) + rindexer (2G) + OS. Recommended for production. |
| `t3.large` | 2 | 8 GB | ~$67 | Heavy indexing, multiple contracts |
| `t3.xlarge` | 4 | 16 GB | ~$133 | Large-scale backfills |

**Rule of thumb:** `ec2_indexer_mem_limit` + `ec2_rpc_proxy_mem_limit` + 1 GB (OS) should not exceed instance RAM.

### RDS PostgreSQL

| Instance | vCPU | RAM | Monthly Cost | Use Case |
|----------|------|-----|-------------|----------|
| `db.t4g.micro` | 2 | 1 GB | ~$13 | **Default.** Dev/staging |
| `db.t4g.small` | 2 | 2 GB | ~$26 | Small production |
| `db.t4g.medium` | 2 | 4 GB | ~$52 | Medium production |
| `db.r6g.large` | 2 | 16 GB | ~$175 | Heavy read workloads |

### Memory Limits (Docker)

| Component | Default | Recommended Minimum | Heavy Use |
|-----------|---------|-------------------|-----------|
| eRPC | `1g` | `512m` | `2g` |
| rindexer | `2g` | `1g` | `4g` |

Increase `ec2_indexer_mem_limit` if rindexer is OOM-killed during large backfills. Check with `docker stats` on the instance.
