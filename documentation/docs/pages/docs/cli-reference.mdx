# CLI Reference

Use this page as the command reference for `evm-cloud`.

For installation, see [Install CLI](/docs/install).

## Global Syntax

```bash
evm-cloud [--color auto|always|never] <command> [flags]
```

## Commands

### `init`

Initialize Terraform and/or scaffold project files.

```bash
evm-cloud init [--dir <path>] [--example <name>] [--list-examples] [--mode easy|power] [--force] [--allow-raw-terraform] [--non-interactive --config <file>] [--skip-terraform-init] [-- <terraform args>]
```

Key behavior:
- Scaffold-first for new directories
- Existing project: init-only unless `--force`
- `--config` on existing project requires `--force` (fails fast otherwise)
- Supports both answers TOML and runtime `evm-cloud.toml` in non-interactive mode
- `--example <name>` bootstraps from local `examples/` when in a repo checkout, otherwise downloads examples from GitHub (current CLI tag, fallback `main`) before init
- `--list-examples` prints canonical example names and aliases
- `--allow-raw-terraform` is required in raw Terraform roots without easy-mode markers

### `deploy`

Provision infrastructure **and** deploy workloads in a single pipeline. This is the primary command for going from zero to running infrastructure.

```bash
evm-cloud deploy [--dir <path>] [--dry-run] [--auto-approve] [--only infra|app] [--tf-args <args>] [--deploy-timeout <secs>] [--module-name <name>] [--allow-raw-terraform] [--json] [-- <deployer args>]
```

- **Full pipeline** (default): terraform init ŌåÆ plan ŌåÆ confirm ŌåÆ apply ŌåÆ parse handoff ŌåÆ invoke deployer
- `--dry-run` runs Terraform plan only (no apply, no deployer)
- `--auto-approve` skips interactive confirmation (required for non-interactive shells)
- `--only infra` runs terraform only (skip workload deployer)
- `--only app` runs deployer only (reads existing terraform state)
- `--tf-args` passes extra arguments to terraform (repeatable, e.g. `--tf-args='-target=module.x'`)
- `--deploy-timeout` optional outer timeout (seconds) for the deployer phase
- `--json` outputs raw handoff JSON and suppresses pretty output (for CI pipelines)
- `--module-name` overrides the Terraform output module name (default: `evm_cloud`)
- Streams real-time deploy progress with phase labels and status icons (Ō£ö for confirmations, ­¤ø¤ for k3s workloads, Ōø┤’ĖÅ for Docker workloads)

Flag conflicts:
- `--only app` + `--dry-run` / `--auto-approve` / `--tf-args` ŌåÆ error (these control terraform)
- If terraform succeeds but deployer fails, prints a recovery hint: `evm-cloud deploy --only app`

:::warning
**EKS limitation:** The CLI deployer does not currently support `compute_engine = "eks"`. For EKS deployments, use `evm-cloud deploy --only infra` (or `evm-cloud apply`) for the Terraform phase, then deploy workloads manually using the [EKS Helm deployer](./guides/external-deployers.mdx#eks-helm-deployer-deployerseks) shell script.
:::

### `apply`

Terraform plan/apply only ŌĆö escape hatch when you need raw terraform control without workload deployment.

```bash
evm-cloud apply [--dir <path>] [--dry-run] [--auto-approve] [--allow-raw-terraform] [--json] [-- <terraform args>]
```

- `--dry-run` runs Terraform plan (no changes applied)
- `--json` outputs raw handoff JSON and suppresses pretty output (for CI pipelines)
- Interactive mode prompts: `Apply these changes? (y/N)`
- Non-interactive shells require `--auto-approve`
- Terraform output is captured for concise UX; detailed output is written to `.evm-cloud/logs/`

:::tip
For most workflows, use `evm-cloud deploy` instead ŌĆö it runs terraform **and** deploys workloads in one step. Use `apply` only when you need terraform-only control.
:::

### `destroy`

Destroy infrastructure resources.

```bash

```

### `kubectl`

Run kubectl commands against the deployed cluster with automatic kubeconfig resolution.

```bash
evm-cloud kubectl [--dir <path>] [--kubeconfig <path>] [-- <kubectl args>]
```

- Auto-resolves kubeconfig from Terraform state (k3s base64 decode, EKS `update-kubeconfig`)
- Falls back to `kubeconfig.yaml` or `.evm-cloud/kubeconfig.yaml` if present
- `--kubeconfig` overrides auto-resolution

```bash
# Examples
evm-cloud kubectl -- get pods -n evm-cloud
evm-cloud kubectl -- logs -n evm-cloud -l app=rindexer -f
evm-cloud kubectl --dir ./my-project -- get nodes -o wide
```

### `local`

Manage a local dev stack (kind cluster with Anvil, eRPC, ClickHouse, and rindexer).

```bash
evm-cloud local <up|down|status|reset> [OPTIONS]
```

Subcommands:
- `up` ŌĆö Start the local stack
- `down` ŌĆö Tear down the local stack
- `status` ŌĆö Show stack status
- `reset` ŌĆö Tear down, clean data, and restart

Options for `up` and `reset`:
- `--rpc <url>` ŌĆö RPC URL (fork source, or direct upstream with `--mainnet`)
- `--mainnet` ŌĆö Skip Anvil, point eRPC directly at `--rpc` URL
- `--fresh` ŌĆö No fork, use chain_id 31337
- `--persist` ŌĆö Persist ClickHouse data across restarts
- `--force` ŌĆö Force-recreate cluster even if it exists
- `--profile <name>` ŌĆö Resource profile: `default` or `heavy`
- `--config-dir <path>` ŌĆö Path to config directory or `rindexer.yaml`
- `--post-deploy <script>` ŌĆö Script to run after stack is healthy

See [Local Dev Stack guide](./guides/local-dev.mdx) for full usage.

## Init Wizard

When running `evm-cloud init` interactively (without `--config` or `--example`), the wizard prompts for:

| Step | Question | Condition | Options | Default |
|------|----------|-----------|---------|---------|
| 1 | Integration mode | Always | `easy`, `power` | `easy` |
| 2 | Project name | Always | free text | `evm-cloud-demo` |
| 3 | Infrastructure provider | Always | `aws`, `bare_metal` | `aws` |
| 4 | Compute engine | Always | AWS: `ec2`, `eks`, `k3s`; Bare metal: `k3s`, `docker_compose` | `ec2` / `k3s` |
| 5 | Workload mode | EC2 or Docker Compose | `terraform`, `external` | `terraform` |
| 6 | AWS region | AWS only | free text | `us-east-1` |
| 7 | Database profile | Always | `byodb_clickhouse`, `byodb_postgres`, `managed_rds`, `managed_clickhouse` | `byodb_clickhouse` |
| 8 | Chains | Always | multi-select: `ethereum`, `polygon`, `arbitrum`, `base`, `optimism`, `hyperliquid` | ŌĆö |
| 9 | RPC endpoint per chain | Always | URL per selected chain | ŌĆö |
| 10 | Indexer config | Always | `generate starter rindexer.yaml`, `use existing path` | generate |
| 11 | Generate eRPC config | Always | yes / no | yes |
| 12 | Instance type | AWS only | free text | `t3.small` |
| 13 | Ingress mode | Always | `none`, `cloudflare`, `caddy`, `ingress_nginx` | `none` |
| 14 | Hostname | Ingress ŌēĀ `none` | free text | ŌĆö |
| 15 | TLS email | `caddy` or `ingress_nginx` | email address | ŌĆö |

After answering, the wizard generates `evm-cloud.toml` (easy mode) or Terraform files (power mode), runs `terraform init`, `terraform fmt`, and `terraform validate`.

For non-interactive use (CI), provide answers via `--config <answers.toml>` with `--non-interactive`.

## Project Modes

Commands route based on marker/root detection:
- `.evm-cloud/mode` (`easy` / `power`) is authoritative
- Without marker:
  - `main.tf` + `versions.tf` => raw/power path
  - `evm-cloud.toml` => easy path
- Both TOML + explicit root without marker => explicit error
