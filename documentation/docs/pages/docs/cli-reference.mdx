# CLI Reference

Use this page as the command reference for `evm-cloud`.

For installation, see [Install CLI](/docs/install).

## Global Syntax

```bash
evm-cloud [--color auto|always|never] <command> [flags]
```

## Commands

### `init`

Initialize Terraform and/or scaffold project files.

```bash
evm-cloud init [OPTIONS] [-- <terraform args>]

  -d, --dir <path>              Project directory (default: .)
  --example <name>              Bootstrap from an example template
  --list-examples               Print available example names and aliases
  --mode <easy|power>           Integration mode
  --force                       Overwrite existing project files
  --allow-raw-terraform         Allow init in raw Terraform roots without easy-mode markers
  --non-interactive             Non-interactive mode (requires --config)
  --config <path>               Answers file for non-interactive init
  --skip-terraform-init         Skip terraform init after scaffolding
```

Key behavior:
- Scaffold-first for new directories
- Existing project: init-only unless `--force`
- `--config` on existing project requires `--force` (fails fast otherwise)
- Supports both answers TOML and runtime `evm-cloud.toml` in non-interactive mode
- `--example` bootstraps from local `examples/` when in a repo checkout, otherwise downloads from GitHub (current CLI tag, fallback `main`)
- `--list-examples` prints canonical example names and aliases

### `deploy`

Provision infrastructure **and** deploy workloads in a single pipeline. This is the primary command for going from zero to running infrastructure.

```bash
evm-cloud deploy [OPTIONS] [-- <deployer args>]

  -d, --dir <path>              Project directory (default: .)
  --dry-run                     Run Terraform plan only (no apply, no deployer)
  --auto-approve                Skip interactive confirmation (required for CI)
  --only <infra|app>            Run only one phase of the pipeline
  --tf-args <args>              Extra arguments for Terraform (repeatable)
  --deploy-timeout <secs>       Timeout in seconds for the deployer phase
  --module-name <name>          Terraform output module name (default: evm_cloud)
  --allow-raw-terraform         Allow deploy in raw Terraform roots
  --json                        Raw handoff JSON output, suppress pretty output
```

- **Full pipeline** (default): terraform init → plan → confirm → apply → parse handoff → invoke deployer
- `--only infra` runs terraform only (skip workload deployer)
- `--only app` runs deployer only (reads existing terraform state)
- Streams real-time deploy progress with phase labels and status icons

Flag conflicts:
- `--only app` + `--dry-run` / `--auto-approve` / `--tf-args` → error (these control terraform)
- If terraform succeeds but deployer fails, prints a recovery hint: `evm-cloud deploy --only app`

:::warning
**EKS limitation:** The CLI deployer does not currently support `compute_engine = "eks"`. For EKS deployments, use `evm-cloud deploy --only infra` (or `evm-cloud apply`) for the Terraform phase, then deploy workloads manually using the [EKS Helm deployer](./guides/external-deployers.mdx#eks-helm-deployer-deployerseks) shell script.
:::

### `apply`

Terraform plan/apply only — escape hatch when you need raw terraform control without workload deployment.

```bash
evm-cloud apply [OPTIONS] [-- <terraform args>]

  -d, --dir <path>              Project directory (default: .)
  --dry-run                     Run Terraform plan only (no changes applied)
  --auto-approve                Skip interactive confirmation (required for CI)
  --allow-raw-terraform         Allow apply in raw Terraform roots
  --json                        Raw handoff JSON output, suppress pretty output
```

- Interactive mode prompts: `Apply these changes? (y/N)`
- Terraform output is captured for concise UX; detailed output is written to `.evm-cloud/logs/`

:::tip
For most workflows, use `evm-cloud deploy` instead — it runs terraform **and** deploys workloads in one step. Use `apply` only when you need terraform-only control.
:::

### `destroy`

Destroy infrastructure resources.

```bash
evm-cloud destroy [OPTIONS] [-- <terraform args>]

  -d, --dir <path>              Project directory (default: .)
  --yes                         Confirm destruction (required)
  --auto-approve                Skip interactive Terraform confirmation
  --allow-raw-terraform         Allow destroy in raw Terraform roots
```

### `status`

Live infrastructure health dashboard. Detects the compute engine, probes running services, and prints an actionable status report with copy-pasteable connection commands.

```bash
evm-cloud status [OPTIONS]

  -d, --dir <path>              Project directory (default: .)
  --module-name <name>          Terraform output module name (default: evm_cloud)
  --json                        Machine-readable JSON output to stdout
  --ssh-key <path>              Override SSH key path (auto-resolved from tfvars)
  --ssh-user <user>             Override SSH user (default: ec2-user for EC2, ubuntu for others)
  --ssh-port <port>             Override SSH port (default: 22)
  --allow-raw-terraform         Allow status in raw Terraform roots
```

Key behavior:
- **Adaptive probing**: automatically detects compute engine from the deployment handoff and uses the right probe strategy
  - **EC2 / Docker Compose / Bare Metal** → SSH into host, check `docker compose ps`, curl health endpoints
  - **K3s / EKS** → auto-generate kubeconfig, check `kubectl get deployments` and pod status
- **Service-aware**: only probes services present in the deployment (eRPC, ClickHouse, Postgres, Caddy, Grafana, Prometheus)
- **Actionable output**: failing services include remediation hints with exact commands to run
- **Exit codes**: `0` = all healthy, `1` = any service down, `2` = any service degraded (useful for CI/scripts)

**EC2 / Docker Compose example output:**

```
 evm-cloud status — my-project (ec2 · us-east-1)

  Services
  ────────────────────────────────────────────────────────

  ✓ rindexer       UP         running · port 18080
  ✓ eRPC           UP         running · https://erpc.example.com

     SSH       ssh -i ~/.ssh/key.pem ec2-user@34.x.x.x
     Logs      aws logs tail /ecs/my-project --follow

```

**K3s example output:**

```

     Cluster        3 nodes · Ready
     Namespace      my-project

  Services
  ────────────────────────────────────────────────────────
  ✓ rindexer       UP         1/1  my-project-indexer-7f8b9 · Running
```

**`--json` mode** outputs a `StatusReport` JSON object to stdout (human output goes to stderr), useful for CI pipelines and monitoring integrations.

:::tip
SSH key and user are auto-resolved from your `secrets.auto.tfvars` and `terraform.tfvars` files. Use `--ssh-key`, `--ssh-user`, and `--ssh-port` only when you need to override the defaults.
:::

### `logs`

Logs command placeholder.

```bash
evm-cloud logs
```

Current behavior: prints `logs command not yet implemented`.

### `kubectl`

Run kubectl commands against the deployed cluster with automatic kubeconfig resolution.

```bash
evm-cloud kubectl [OPTIONS] <kubectl args>

  -d, --dir <path>              Project directory (default: .)
  --kubeconfig <path>           Override kubeconfig path (skips auto-resolution)
```

- Auto-resolves kubeconfig from Terraform state (k3s base64 decode, EKS `update-kubeconfig`)
- Falls back to `kubeconfig.yaml` or `.evm-cloud/kubeconfig.yaml` if present

```bash
# Examples
evm-cloud kubectl get pods -n evm-cloud
evm-cloud kubectl logs -n evm-cloud -l app=rindexer -f
evm-cloud kubectl --dir ./my-project get nodes -o wide
```

### `local`

Manage a local dev stack (kind cluster with Anvil, eRPC, ClickHouse, and rindexer).

#### `local up`

Start the local stack.

```bash
evm-cloud local up [OPTIONS]

  --rpc <url>            RPC URL (fork source, or direct upstream with --mainnet)
  --mainnet              Skip Anvil, point eRPC directly at --rpc URL
  --fresh                No fork, chain_id=31337
  --profile <name>       Resource profile: default, heavy (default: default)
  --persist              Persistent ClickHouse data across restarts
  --force                Force-recreate cluster even if it exists
  --config-dir <path>    Path to config directory or rindexer.yaml
  --post-deploy <path>   Script to run after stack is healthy
```

#### `local down`

Tear down the local stack.

```bash
evm-cloud local down
```

#### `local status`

Show stack status (pods, endpoints, resource usage).

```bash
evm-cloud local status
```

#### `local reset`

Tear down, clean persistent data, and restart. Accepts the same options as `up`.

```bash
evm-cloud local reset [OPTIONS]
```

See [Local Dev Stack guide](./guides/local-dev.mdx) for full usage.

## Init Wizard

When running `evm-cloud init` interactively (without `--config` or `--example`), the wizard prompts for:

| Step | Question | Condition | Options | Default |
|------|----------|-----------|---------|---------|
| 1 | Integration mode | Always | `easy`, `power` | `easy` |
| 2 | Project name | Always | free text | `evm-cloud-demo` |
| 3 | Infrastructure provider | Always | `aws`, `bare_metal` | `aws` |
| 4 | Compute engine | Always | AWS: `ec2`, `eks`, `k3s`; Bare metal: `k3s`, `docker_compose` | `ec2` / `k3s` |
| 5 | Workload mode | EC2 or Docker Compose | `terraform`, `external` | `terraform` |
| 6 | AWS region | AWS only | free text | `us-east-1` |
| 7 | Database profile | Always | `byodb_clickhouse`, `byodb_postgres`, `managed_rds`, `managed_clickhouse` | `byodb_clickhouse` |
| 8 | Chains | Always | multi-select: `ethereum`, `polygon`, `arbitrum`, `base`, `optimism`, `hyperliquid` | — |
| 9 | RPC endpoint per chain | Always | URL per selected chain | — |
| 10 | Indexer config | Always | `generate starter rindexer.yaml`, `use existing path` | generate |
| 11 | Generate eRPC config | Always | yes / no | yes |
| 12 | Instance type | AWS only | free text | `t3.small` |
| 13 | Ingress mode | Always | `none`, `cloudflare`, `caddy`, `ingress_nginx` | `none` |
| 14 | Hostname | Ingress ≠ `none` | free text | — |
| 15 | TLS email | `caddy` or `ingress_nginx` | email address | — |

After answering, the wizard generates `evm-cloud.toml` (easy mode) or Terraform files (power mode), runs `terraform init`, `terraform fmt`, and `terraform validate`.

For non-interactive use (CI), provide answers via `--config <answers.toml>` with `--non-interactive`.

## Project Modes

Commands route based on marker/root detection:
- `.evm-cloud/mode` (`easy` / `power`) is authoritative
- Without marker:
  - `main.tf` + `versions.tf` => raw/power path
  - `evm-cloud.toml` => easy path
- Both TOML + explicit root without marker => explicit error

### The `.evm-cloud/` Directory

In Easy mode, `evm-cloud deploy` calls `prepare_workspace` which reads `evm-cloud.toml` and generates a bridge directory:

```
.evm-cloud/
├── mode                    # "easy" or "power" — authoritative mode marker
├── main.tf                 # Generated Terraform root calling the evm-cloud module
├── variables.tf            # Generated variable declarations
├── outputs.tf              # Generated output declarations
├── terraform.auto.tfvars.json  # Generated from evm-cloud.toml values
├── config-bundle/          # Assembled config files for the deployer
│   ├── erpc.yaml
│   ├── rindexer.yaml
│   ├── abis/
│   └── .env                # Auto-generated from secrets.auto.tfvars (see below)
├── logs/                   # Terraform plan/apply log files
└── .terraform/             # Terraform provider cache
```

You should `.gitignore` the entire `.evm-cloud/` directory — it's generated on every `deploy`.

## `evm-cloud.toml` Schema (Easy Mode)

Easy mode uses `evm-cloud.toml` as a simplified configuration file. The CLI reads it and generates Terraform files in `.evm-cloud/`.

```toml
schema_version = 1

[project]
name = "my-indexer"
region = "us-east-1"              # optional, default: us-east-1
deployment_target = "managed"      # optional: managed, hybrid, self_hosted
runtime_arch = "multi"             # optional: amd64, arm64, multi

[compute]
engine = "ec2"                     # ec2, eks, k3s, docker_compose
instance_type = "t3.small"         # optional
workload_mode = "terraform"        # optional: terraform, external

[compute.ec2]                      # optional, EC2-specific overrides
rpc_proxy_mem_limit = "1g"
indexer_mem_limit = "2g"
secret_recovery_window_in_days = 7

[compute.k3s]                      # optional, k3s-specific overrides
version = "v1.30.4+k3s1"

[database]
mode = "managed"                   # managed or self_hosted
provider = "aws"                   # aws or bare_metal
storage_backend = "clickhouse"     # clickhouse or postgres

[indexer]
config_path = "./rindexer.yaml"
erpc_config_path = "./erpc.yaml"   # optional
chains = ["ethereum", "base"]

[rpc]
[rpc.endpoints]
ethereum = "https://eth.llamarpc.com"
base = "https://base.llamarpc.com"

[ingress]
mode = "none"                      # none, cloudflare, caddy, ingress_nginx
domain = "rpc.example.com"         # required when mode != none
tls_email = "admin@example.com"    # required for caddy or ingress_nginx

[secrets]
mode = "provider"                  # inline, provider, external

[networking]                       # optional
vpc_cidr = "10.42.0.0/16"
enable_vpc_endpoints = false
environment = "dev"

[postgres]                         # optional, when storage_backend = postgres
instance_class = "db.t4g.micro"
engine_version = "16.4"

[containers]                       # optional
rpc_proxy_image = "ghcr.io/erpc/erpc:latest"
indexer_image = "ghcr.io/joshstevens19/rindexer:latest"

[monitoring]                       # optional
enabled = false
grafana_hostname = "grafana.example.com"

[bare_metal]                       # optional, when provider = bare_metal
rpc_proxy_mem_limit = "1g"
indexer_mem_limit = "2g"

[streaming]                        # optional
mode = "disabled"
```

All fields marked `optional` can be omitted — the CLI uses sensible defaults. Required fields are `[project].name`, `[compute].engine`, `[database].*`, `[indexer].*`, `[rpc].endpoints.*`, `[ingress].mode`, and `[secrets].mode`.

## Deploy Internals

### Deploy Lock

The `deploy` command acquires an exclusive file lock (`.evm-cloud-deploy.lock`) before running. This prevents concurrent deploys from corrupting Terraform state.

- The lock file contains the PID and timestamp of the holder
- If a previous deploy crashed (stale lock from a dead PID), the CLI auto-recovers
- If another deploy is running, the CLI exits with `DeployLockBusy`

### `workload_mode = "terraform"` Skip Behavior

When `workload_mode = "terraform"`, Terraform provisioners deploy workloads during `terraform apply` (e.g., Docker Compose services via cloud-init or SSH provisioner). In this case, `evm-cloud deploy` skips the external deployer phase after apply — workloads are already running.

To force a re-deploy of workloads (e.g., config change without infra change), use `--only app`.

### Auto `.env` Generation

During deploy, the CLI automatically generates a `.env` file in the config bundle directory from your `secrets.auto.tfvars`. This injects database credentials and RPC URLs into the deployer without manual wiring:

| Source Variable | Generated `.env` Key | Condition |
|----------------|---------------------|-----------|
| `indexer_clickhouse_url` | `CLICKHOUSE_URL` | ClickHouse backend |
| `indexer_clickhouse_password` | `CLICKHOUSE_PASSWORD` | ClickHouse backend |
| `indexer_clickhouse_user` | `CLICKHOUSE_USER` | ClickHouse backend (default: `default`) |
| `indexer_clickhouse_db` | `CLICKHOUSE_DB` | ClickHouse backend (default: `rindexer`) |
| `indexer_postgres_url` | `DATABASE_URL` | Postgres backend |
| eRPC internal URL | `RPC_URL` | When `rpc_proxy_enabled = true` |

The `.env` file is written with `0600` permissions (owner-only read/write).

### SSH Variable Resolution

The CLI resolves `ssh_private_key_path` from your tfvars files automatically for deployers that need SSH access (compose deployer for EC2/Docker Compose, k3s deployer for k3s). It checks `secrets.auto.tfvars` and `terraform.tfvars` in order.

## Version Pinning

Create a `.evm-cloud-version` file in your project root (or any parent directory) to pin the CLI version:

```bash
echo "0.0.1-alpha5" > .evm-cloud-version
```

When this file exists, the CLI checks its version on every command. If there's a mismatch, it exits with an error:

```
Error: CLI version 0.0.1-alpha5 does not match pinned version 0.0.2 in .evm-cloud-version
```

This prevents accidental version drift across team members or CI environments. The `v` prefix is optional (`v0.0.1-alpha5` and `0.0.1-alpha5` are equivalent).
